Add a **completeness term** to the envelope floor — `executedTests = passed + failed + errored` (**excludes** `SKIPPED`) — and a **new operational error `NO_TESTS_RUN`** (+ hint) when a fresh report exists but `executedTests == 0`. | Closes the vacuous-green class found in adversarial review. Distinct from `REPORT_NOT_PRODUCED` (no report at all) — here a report exists but is empty. Lives in the application floor, so the frozen `NormalizedRun.ok()` is untouched. An all-`SKIPPED` run therefore does **not** silently green. |
| D30 | **Agent-facing envelope field = `failures[]` (carries `Finding`); domain record field = `NormalizedRun.findings`.** The test-failure Envelope exposes a top-level **`failures[]`** array (the established agent-facing term — CONTEXT.md, tool-catalog.md, DESIGN §6), each element a `Finding` serialized with the frozen `@JsonTypeInfo(use=NAME, property="kind")` discriminator so the agent branches `TestFinding` vs `ContainerFinding` on `kind`. The frozen **domain** field stays `NormalizedRun.findings` (ADR-0007). | Resolves a corpus naming ambiguity (`failures[]` at the envelope layer vs `findings` in the frozen record) so the `jq -e` acceptance assertions and the agent's branch on `kind` are authorable. No frozen field is renamed; only the envelope↔domain mapping is pinned. |

## PRD-2 — "remove-Bash-on-self" (grill 2026-06-04, in progress)

Decisions for the next phase: **v1 catalog completion toward the product thesis**, not the PRD-1
architecture. Scope per D32. Status: grill in progress, **not yet implemented**.

| # | Decision | Rationale |
|---|---|---|
| D31 | **PRD-2 sequencing axis = value-first ("remove-Bash-on-self"), not risk-first.** PRD-1 already proved the entire foundation (frozen schema ADR-0007, dispatch flow, `Envelope`, run-cache, `get_log`, `CommandExecutorPort`); everything remaining is **additive/leaf** — Node/Go are more normalizers behind a proven port, forge is a self-contained 2nd port, native is terminal packaging. **Nothing foundational-risky remains**, so optimize for the product thesis instead of de-risking. | The product delivers **zero** value until the Bash permission can actually be removed; `run_tests` alone does not get there. With no risky base left to de-risk-before-building-atop, the highest-value next step is the smallest slice that realizes the thesis and unlocks dogfooding-on-self (which surfaces real empirical risk). |
| D32 | **PRD-2 scope = minimal dev-loop: git read-only (5 verbs) + Maven `build` + bootstrap skill.** Removes Bash for `edit → build → test → inspect-git`. **Out of scope:** `gh` and **mutating** git stay on the **transitional dangerous-git deny-list** (D15/D16) — Bash-for-git-mutation persists until post-v1 native mutating-git verbs. Forge (`pr_*`) → PRD-3; `install`/`lint`/`run_task` → later. | KISS/YAGNI: the smallest slice that delivers the remove-Bash thesis for *this repo's own* loop. Forge `pr_*` is read-only and does not cover issue/PR **create** (what `gh`/orchestrate use), so it does not help remove Bash from the dev loop — it earns its own PRD. `install` is vacuous for Maven (D21); `run_task` is the highest-security verb (G14) — both add surface without advancing the thesis. |
| D33 | **`build` compile-error output = new build-specific `CompileDiagnostic{file, line, col, severity, message}`, carried as envelope `diagnostics[]`** (distinct from test `failures[]`), parsed from maven-compiler-plugin `[ERROR] <file>:[<line>,<col>] <msg>` console lines; full compiler output retained via `handle`/`get_log`. The frozen ADR-0007 test schema (`Finding`/`SourceRef`) is **untouched**. → candidate **ADR-0009**. | A compile diagnostic has **no test identity** and no test `Outcome` (PASSED/SKIPPED are meaningless), and frozen `SourceRef` carries **no column** — folding it into `ContainerFinding(FILE)` would force `outcome=ERRORED`, drop the column, and stretch a schema ADR-0007 declared to be about *test results*. Console-parse is unavoidable (compile failures produce **no report-file**, unlike tests), but `file:[line,col]` coordinates are **locale-independent** (only the message text is localized → passed through as untrusted content), so D8's "no fragile stdout scraping" concern is contained. Rejected: reuse `ContainerFinding` (above); tracer-with-no-parser (loses the `file:line` differential that justifies `build` over removed Bash). |
| D34 | **git read-only output = rich per-verb schemas parsed from `git --porcelain=v2` / `--format=`.** `git_status`/`git_branch` → small inline structured shapes; `git_log` → capped commit list (`sha`, `short`, `author`, `dateIso`, `subject`; default last-N) with full body via `git_show`; `git_diff`/`git_show` → inline `files[]` (`path`, `added`, `deleted`, `status`) + the patch behind a `handle`/`get_log` (consistent with the already-decided `git_diff` handle reuse). `manager` is null for git verbs (CONTEXT.md). | Sufficiency-first + token-efficient + locale-stable: porcelain / `--format` is git's machine contract — the project's "parse the machine format, never scrape stdout" rule (D8) applied to git. The highest-volume verbs (`git_log`/`git_diff`, 773-call evidence) get a handle boundary so large patches never flood the envelope. Rejected: thin-structured (agent re-parses author/date/ahead-behind → weakens sufficiency) and passthrough-porcelain (pushes all parsing back to the agent → contradicts agent-first). |
| D35 | **Transitional deny-list posture = "block destruction only."** The bootstrap's dangerous-git deny-list **denies** irreversible / history-rewriting / working-tree-loss ops and **allows** forward-progress git. DENY: `push --force`/`-f`/`--force-with-lease`, `push --delete`/`push :ref`, `reset --hard`, `rebase`, `commit --amend`, `filter-branch`/`filter-repo`, `clean -f`/`-fd`/`-fdx`, `checkout`/`restore -- <path>`, `checkout -f`, `branch -D`/`-d`, `tag -d`, `update-ref -d`, `stash drop`/`clear`, `reflog expire`, `gc --prune=now`, `prune`, `worktree remove --force`, `submodule deinit`. ALLOW: `add`, `commit`, `push` (plain), `branch <new>`, `switch`/`checkout <branch>`, `merge --ff`, `fetch`, `pull`. | Goal = the agent cannot **destroy** the repo, yet keeps its `edit → commit → push` loop (required for dogfooding-on-self). Accepts the **known G6 fragility** — a denylist cannot catch `git -c <config-injection>`, aliases, or command-chaining — because it is an explicitly **transitional bridge**; the real guarantee arrives when mutating-git becomes allowlist MCP verbs (post-v1 roadmap). Rejected: also-block-push (breaks this repo's push-driven flow, defers publish to a human) and block-all-mutation (agent loses the entire git write loop, contradicting "remove Bash without losing the loop"). |
| D36 | **Empty-but-initialized repo (unborn HEAD, no commits) maps to `ok=true` + empty result in `git_log` and `git_diff` (Option A).** The blunt exit-code floor (`exitCode != 0 → NOT_A_GIT_REPOSITORY`) is replaced with a two-probe discriminator on the error path only: `git rev-parse --is-inside-work-tree` (exit 0 → inside a work tree) plus `git rev-parse --verify --quiet HEAD` (non-zero → no commits yet); both true → ok-empty (`git_log` returns `commits:[]`; `git_diff` returns `gitDiff:[]` with an empty stashed handle). A genuine non-repo still returns `NOT_A_GIT_REPOSITORY` (false-green floor preserved). `git_status` and `git_branch` are unaffected (they already exit 0 on unborn HEAD). `git_show` is out of scope — an unresolvable HEAD ref on an empty repo is a genuine `COMMIT_NOT_FOUND`. | **Option A ("ok-empty read")** matches the behavior of `git_status`/`git_branch` (already correct) and is semantically accurate: an empty `git init`-ed directory IS a git repository — it merely has no history to enumerate. **Rejected: Option B (`EMPTY_REPOSITORY` error code)** — a new error code is YAGNI; agents querying an unborn-HEAD repo need an empty list, not a new branch on an error code. **Rejected: keeping `NOT_A_GIT_REPOSITORY`** — factually wrong (the path IS inside a work tree), misleads the agent into thinking no repository exists. |

## Distribution & release (grill 2026-06-05)

Decisions locking **how the native image is shipped** to a harness — the npm/npx launcher channel,
the topology, the v1 platform matrix, uncovered-platform handling, signing, version pinning, build
provenance, and the deferred secondary channels (plus the accepted Node-shim footprint cost).
Architecture detail (the publish → npm-resolve → shim-spawn flow) lives in
**[ADR-0010](../adr/0010-npm-launcher-distribution.md)**; the new traps are **G17–G19**.

| # | Decision | Rationale |
|---|---|---|
| D37 | **Primary distribution = npm/npx launcher (D-CHANNEL).** The Bootstrap skill writes `.mcp.json` to invoke `command="npx", args=["-y","no-bash-mcp@<EXACT_VERSION>"]` (the esbuild/biome/turbo model). Honest tension (stated, not hidden): native is sold as "runs without a runtime installed", yet npm reintroduces **Node** — for **install, not execution**; Node is near-certain present because Claude Code itself ships via npm, and execution remains a pure native process. → **[ADR-0010](../adr/0010-npm-launcher-distribution.md)**. | npm `os`/`cpu` selects the exact OS×arch binary with a precision `platform_overrides` cannot give (D44); tamper-evidence comes **free** from the npm registry's tarball hash (`dist.integrity`, SHA-512) + the exact version pin (D42) — the npx model has **no committed lockfile**, so reproducibility rests on exact pins, not a checked-in `package-lock.json`; `.mcp.json` is a pure config-write the Bootstrap skill **already** does (no new step), matching the user's npx-transparency anchor. **Rejected: bundle all platforms in one fat package** (ships unused binaries to every install — the per-platform topology in D38 selects exactly one). |
| D38 | **Topology = `optionalDependencies`-only, ZERO runtime download (D-TOPOLOGY).** A **thin JS launcher** package `no-bash-mcp` (small shim) declares per-platform **scoped** packages `@no-bash-mcp/<os>-<arch>` as `optionalDependencies`, each carrying exactly **one** native binary and gated by npm `os`/`cpu`. npm installs only the matching package; the shim resolves it and spawns it, **forwarding stdio**. **No `postinstall` network fetch.** No platform resolves → the shim **fails clear** (names the platform, points at the deferred secondary channel). **Publish order: all platform packages BEFORE the launcher** so its `optionalDependencies` resolve (race avoidance). New trap **G18**. | **Rejected: `postinstall` network download** (the original esbuild model) — esbuild **moved off** it precisely because it is fragile (`--ignore-scripts`, corporate proxies, airgapped CI) **and** a supply-chain vector (an unverified fetch executed at install time); reintroducing it would contradict a tool whose whole thesis is **removing** a dangerous permission. |
| D39 | **v1 platform matrix = the 4 tuples GraalVM native-image for JDK 25 supports: `linux-x64`, `linux-arm64` (aarch64), `darwin-arm64` (Apple Silicon), `win32-x64` (D-MATRIX).** This is **dictated by the toolchain**, not a judgement call. Each tuple's GraalVM JDK-25 native support is **verified in CI before being promised** (extends G15). | Promising only what the toolchain can actually produce keeps the channel honest and CI-verifiable (source: GraalVM JDK-25 release notes). **Rejected: promise `darwin-x64`/`win32-arm64` from a best-effort build** — no supported toolchain (`win32-arm64` has none; `darwin-x64`/Intel is deprecated upstream, 25.0.1 the last release, future = arm64-only) → would ship an unverified or non-existent binary, the opposite of fail-clear. |
| D40 | **Uncovered platforms (`win32-arm64`, `darwin-x64`/Intel) = FAIL-CLEAR (D-UNCOVERED).** The shim errors actionably (the fail-clear mechanism of D38) rather than doing nothing silently; `win32-arm64` can run the `win32-x64` build under Windows emulation meanwhile. A JVM-jar fallback is **deferred to the roadmap** as an explicit escape hatch. See G18. | Fail-clear over silent breakage keeps the channel honest. **Rejected: ship a JVM-jar fallback now** for the uncovered platforms — it would reintroduce the JRE dependency the native image exists to kill; deferred (YAGNI until evidence of a real user stranded on an uncovered platform). |
| D41 | **macOS `darwin-arm64` AD-HOC codesign (`codesign -s -`, free) in CI, applied AFTER all post-processing (strip etc.), is MANDATORY (D-SIGNING).** Ad-hoc is **sufficient for the npm channel** because npm-extracted files do **not** carry the `com.apple.quarantine` bit, so Gatekeeper's **notarization** gate never fires. **PAID signing** (Apple Developer ID + notarization ~$99/yr; Windows OV/EV cert) is **DEFERRED**, coupled to the secondary channels (D44): only `.mcpb`/`curl`-download set MOTW/quarantine and thus require notarization; an unsigned browser-delivered `.exe` triggers SmartScreen (EV suppresses). New trap **G17**. | Apple Silicon enforces that **all native ARM64 code must be signed or the OS prevents execution** (users cannot bypass); a linker-only/corrupt signature gets **SIGKILL**. Smoking gun: **OpenAI codex CLI** (May 2026, `openai/codex#21199`) installs via `npm install -g` yet its Darwin ARM64 binary **fails to spawn** ("Unknown system error -88") — proving the **npm channel does not exempt you from arm64 signing**. **Rejected: rely on GraalVM/linker auto-signing** — it can be the corrupt linker-only signature that gets SIGKILL, or be **invalidated by later post-processing** (strip); hence the explicit ad-hoc step **after** post-proc. **Rejected: do paid notarization now** — unnecessary for npm (no quarantine bit) and only needed once the MOTW-setting secondary channels ship. |
| D42 | **The Bootstrap skill writes an EXACT version pin (`npx -y no-bash-mcp@1.4.2`), never a float (`@latest`) (D-PIN).** Updates are an **explicit action** (re-run bootstrap / bump the pin) — reproducible and auditable. | A tool whose thesis is **removing a dangerous permission** must **not** silently auto-update its own security-critical binary. **Rejected: `@latest` / a semver float** — would let the registry change the executing binary out from under the harness with no audit trail, exactly the kind of invisible drift this tool exists to eliminate. |
| D43 | **Publish all packages with `npm --provenance` (GitHub Actions OIDC → Sigstore) for a verifiable build attestation** (which repo / commit / workflow produced the artifact) — free (D-PROVENANCE). This is the **ORIGIN** layer atop the registry tarball's tamper-evidence; composes with the mpak MCP trust framework later. | High-value / low-cost for a security tool: the registry tarball hash proves the bytes **were not tampered with** but says nothing about **where they came from**; provenance closes that gap. **Rejected: registry tarball integrity alone** — gives tamper-evidence but **not origin**, leaving "is this artifact really from the no-bash-mcp repo's CI?" unanswerable; provenance is free via Actions OIDC, so omitting it is unjustifiable for a security tool. |
| D44 | **Secondary channels (`.mcpb`, `curl\|sh` / Homebrew / Scoop) are DEFERRED to the roadmap (D-SECONDARY).** `.mcpb` (MCP Bundle, ex-DXT) is a parallel desktop channel but **not v1-primary**: its manifest has **zero integrity/signing fields** (needs external trust e.g. mpak — weak for a security tool); `platform_overrides` is keyed by `process.platform` (`win32`/`darwin`/`linux`) **not by arch** (no `${arch}` variable), making per-arch packaging awkward; and `.mcpb` sets MOTW/quarantine → forces the **paid notarization** deferred in D41. `curl\|sh`/brew/Scoop PATH-installers are likewise deferred (truest to the zero-runtime-dep native thesis, but need a discrete install step **and** the paid signing). New trap **G19**. | KISS/YAGNI: one well-understood primary channel (D37/D38) ships v1; the secondaries each need paid signing and/or external trust they cannot yet provide. **Rejected: ship `.mcpb` / `curl\|sh` as a v1-primary channel now** — each reintroduces the paid-signing obligation (MOTW/quarantine, SmartScreen) and, for `.mcpb`, an external trust framework the manifest cannot itself provide; not justifiable for a security tool's v1. |
| D45 | **The npx Node-shim footprint cost is an ACCEPTED, eyes-open trade (D-FOOTPRINT).** The npx model keeps a thin Node shim in front of the native MCP for the whole session (pipes stdio), adding **~30–50 MB RSS** + the Node dependency — this slightly undercuts the "low-RAM native" argument in `build-and-distribution.md` and is **documented as accepted** (esbuild does the same; Node is present anyway). | The shim is the price of npx ergonomics (the version pin D42, os/cpu selection D38, registry-hash integrity D37). **Rejected: have the Bootstrap skill rewrite `.mcp.json` to point directly at the resolved native binary path** (pure-native footprint, no Node shim) — it makes `.mcp.json` **machine-specific / brittle** and **loses npx ergonomics** (the version pin, os/cpu selection, registry-hash integrity all evaporate); the ~30–50 MB shim is the price of those properties. |

## PRD-3 — Node + Go ecosystems (grill 2026-06-05)

Resolves the **PRD-3 numbering collision** (D32 "Forge → PRD-3" vs issue #44 "PRD-003: distribution")
and re-opens the sequencing axis now that the value-first "remove-Bash-on-self" thesis (D31/D32) has
**run out of v1-scoped road**: for this JVM repo the read-only dev loop (`build`/test/git) is already
covered, forge `pr_*` is read-only and (per D32's own caveat) does **not** remove `gh` from the loop,
Node/Go do not touch a JVM repo, and mutating-git + gh-create are post-v1.

| # | Decision | Rationale |
|---|---|---|
| D46 | **PRD-3 = Node + Go ecosystems (`run_tests` across `jest --json` + `go test -json`), NOT distribution.** "v1-complete" is defined as **the universal-schema thesis proven in shipped code across all three dissimilar report formats** — not as an installable-but-JVM-only milestone. Distribution (issue **#44**) is **re-sequenced to PRD-4**; D32's "Forge → PRD-3" is **superseded** (forge becomes a later PRD).
